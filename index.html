Welcome to my railway reservation system website 'Fantasy Express' where you can find the ride of your dreams!

As advised, I have made a few assumptions and modified the structure of my database accordingly. In no particular order, they are as follows:
  • Train Schedule tab allows you to view all records in TrainStatus table.
  • Book Ticket tab implements Booking procedure.
  • Ticket Status tab allows you to view Ticket Status and has the Cancel Ticket option on it a well.
  • Any interface for trains to be added to the TrainStatus table have not been made on the website, but necessary checks to ensure valid records are inserted into TrainStatus table have been added to create table query.
  • For checking weekdays available for a train, an integer value composed of numbers 2 to 6 is used along with DAYOFWEEK(date) function. Note that DAYOFWEEK assumes Sunday = 1, NOT Monday = 1.
  • Although total number of seats booking in each category is defined as 10, it would not make sense to include them in the table so this constraint is ignored and variable values are allowed for each trainnumber in TrainsList, however they have all been instantiated to 10 for now for the purpose of this constraint.
  • The same train may be used to go from a source to a destination and come back from there, so source and destination are kept in TrainStatus table instead of TrainsList table.
  • Since source and destination are now in TrainStatus table, each journey is unique and is now identified by an additional column called trainsstatusid in the TrainStatus table which is also used now in the Passengers table.
  • If a ticket is cancelled, and there exists a next-in-line waiting ticket for the same journey, if they both have different seat types and the seat type of waiting ticket is already booked to the total capacity of that train, the ticket is not changed to confirmed.
  • It is assumed that the description given meant that the total number of tickets in waiting for any train journey must be 2 whether they are for different seat types or the same.
  • Since it was not explicitly mentioned, the input values for booking ticket has been taken together as one whole input field, though processing has been done separately.
  • No password or check is implemented for each ticket so it can be viewed by anyone as this was not mentioned in the problem description.
  • For the same reason, ticket can be cancelled by anyone pressing the button as well
